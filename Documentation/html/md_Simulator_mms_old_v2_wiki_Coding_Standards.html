Coding Standards

This is an incomplete list of some coding standards that I tried to follow when I was coding the simulator.

Line Length

In general, lines should not exceed 100 characters in length. Lines longer than 100 characters should be broken up into two lines.

Constants

Don't use string literals for any value of type Color, Direction, Key, Layout, or InterfaceType. Instead, to provide better safety, get the string value by using the appropriately defined map, as in:

1 {c++}
2 std::string northString = STRING_TO_DIRECTION.at(Direction::NORTH);

Braces

Opening braces should appear on the same line of a class/conditional/function header. Closing braces should appear on their own line. There should a be a single space before every opening brace. Never omit braces.

1 {c++}
2 void foo() {
3 }
4 
5 class Bar {
6 }
7 
8 if (x < 4) {
9 }
10 
11 while (i < myList.size()) {
12 }

Parameter/Argument Lists

Parameter/Argument lists should follow arguments with a comma and single space.

1 {c++}
2 void foo(int one, int two, std::string three) {
3 }

Long parameter/argument lists should be broken into multiple lines. The first part of the second line should be indented four spaces in header files, eight in .cpp files.

1 {c++}
2 // .h file
3 void foo(int one, int two, std::string three, std::string four, long five,
4  bool six, long long seven, std::string eight);
5 
6 // .cpp file
7 void Bar::foo(int one, int two, std::string three, std::string four, long five,
8  bool six, long long seven, std::string eight) {
9 }

Namespaces

User-defined namespaces should be used where appropriate. Namespaces do not force an additional level of indentation.

Namespaces should be fully qualified, though there may be infrequent exceptions to this rule.

Comments

Comments should be used abundantly to describe the intention of the code.

Descriptions of functions, variables, etc. should be placed in the ".h" files, over the thing that they describe

1 {c++}
2 // This does blah blah
3 void foo(int a, int b);

Identifiers

Identifiers should be chosen thoughtfully and carefully. Names should not be abbreviated unless it is abundantly clear what the abbreviation means.

Naming Conventions

Local variables should use camelcase notation

1 {c++}
2 int someVariable;

Functions/Methods should also use camelcase notation

1 {c++}
2 void fooBar();

Classes should also use camelcase, but begin with a capital letter

1 {c++}
2 class GreatClass;

Instance/Member variables should be prefixed with m_

1 {c++}
2 int m_highScore;

Declaration/Initialization

If possible, always intialize variables when you declare them

1 {c++}
2 int myInteger = 0;

Initialization Lists

Initialization lists are always preferred to performing initialization in the body of a class's constructor

1 {c++}
2 Foo::Foo(int one, int two, int three) : m_one(one), m_two(two) {
3  m_three = three; // Don't do this
4 }

Indentation

One level of indentation equals four spaces. No more, no less, and certainly no tab characters.

If you're performing a multi-line assignment, put the equals sign on the first line, as in:

1 {c++}
2 int reallyLongVariableNameThatReallyShouldBeSomethingElse =
3  someReallyLongFunctionNameThatCouldProbablyBeShorter();

Pass by Reference vs Pass by Value vs Passing a Pointer

Pass primitives by value. Pass objects by const reference. Never pass by non-const reference. If an argument needs to be modified from within a function (which shouldn't happen frequently), pass a pointer to the object.

1 {c++}
2 int myFoo(int bar) {
3 }
4 
5 int myBar(const Bar& bar) {
6 }
7 
8 int modBar(Bar* bar) {
9 }

Placement of the Pointer / Placement of the Address-Of Operator

The pointer goes next to the type name

1 {c++}
2 Bar* bar = NULL;
3 
4 int foo(Dog* doggie) {
5 }

The address-of operator goes next to the type name

1 {c++}
2 int foo(const Bar& myBar) {
3 }

Casting

Avoid using casts (it's a sign that the code needs to be cleaned up). However, if you need to cast, avoid using C casts. Instead, use C++ casts.

1 {c++}
2 int bad = (int) getLong(); // Don't do this
3 int good = static_cast<int>(getLong()); // Do this instead

Includes

Includes in the headers files should be organized into two groups: <...> includes (libraries, things we didn't write) and "..." includes (things we wrote). The first group should appear before the second, with exactly one newline separating the groups. The groups should be alphabetized.

1 {c++}
2 #include <iostream>
3 #include <vector>
4 
5 #include "MyFileOne.h"
6 #include "ZooFoo.h"

Includes in cpp files should be organized in the same two groups, with one exception: the header file corresponding to a particular class should appear on its own line before any other includes, as in:

1 {c++}
2 #include "MyClass.h"
3 
4 #include <iostream>
5 #include <vector>
6 
7 #include "MyFileOne.h"
8 #include "ZooFoo.h"

Logging

Always directly follow an error log with a call to quit. Likewise, always make sure to log an error before calling quit.

1 {c++}
2 L()->error("You messed up!");
3 SimUtilities::quit();

When continuing a string onto the next line, use the following format:

1 {c++}
2 L()->warn(
3  "This is a really long log statement, and I think that I'm going"
4  " to put an interger here: %v. And here: %v. And maybe also here:"
5  " %v. OK just one more: %v. I'll keep going, going, going since"
6  " this is such a long log statement",
7  1234,
8  mazeHeight() + mazeWidth(),
9  someReallyLongFunctionName(),
10  3+4+5+6+7+8);

If you have a lot of values to substitute in, consider putting each on their own line. Note that the string continuation begins with a space. Also note that the closing parenthesis appears on the same line as the last argument.

Gotchas

Don't use sim::P() or sim::S() to initialize any static variables. The initialization order of static variables is undefined and thus can mess with the delicate instantiation of the Param and State objects.
